# 静态资源分享网站需求文档（GitHub Pages / 无后端版）

## 1. 项目概述

### 1.1 项目名称

个人资源分享网站

### 1.2 项目定位

本项目是一个部署在类似 GitHub Pages 的静态托管环境中的资源展示网站，主要用于展示和分享个人整理的资源内容，例如软件工具、模板素材、教程文章、网站链接、网盘资源等。

### 1.3 项目目标

- 搭建一个纯前端、可静态部署的个人资源站。
- 让用户能够快速浏览、筛选、搜索和查看资源内容。
- 让站长可以通过修改本地文件或仓库内容的方式维护资源。
- 在没有传统后端、数据库和后台管理系统的前提下，仍保持良好的展示效果和基本可用性。

### 1.4 部署约束

- 网站必须可部署到 GitHub Pages、Cloudflare Pages、Netlify 静态托管或类似平台。
- 网站不能依赖常驻 Node.js 服务。
- 网站不能依赖数据库。
- 网站不能依赖服务端渲染接口。
- 所有页面内容需在构建时生成，或由浏览器直接读取静态 JSON/Markdown 数据。

## 2. 建设范围

### 2.1 本期保留功能

- 首页展示
- 资源分类展示
- 资源详情展示
- 标签筛选
- 前端搜索
- 响应式布局
- SEO 基础配置
- 站点公告与关于页面
- 静态化内容更新流程

## 3. 用户角色

### 3.1 访客

- 浏览首页和资源列表
- 使用搜索和筛选功能
- 查看资源详情
- 点击下载或访问外部资源

### 3.2 站长

- 在本地维护资源数据文件
- 新增、修改、删除资源内容
- 维护分类、标签、公告和页面文案
- 通过 Git 提交并重新部署网站

## 4. 核心使用场景

### 4.1 访客浏览资源

访客进入首页后，通过推荐资源、分类导航或搜索功能找到目标资源，并进入详情页查看说明、截图、标签、版本和下载方式。

### 4.2 访客按分类查找

访客进入分类页后，按照“软件工具”“模板素材”“教程文档”“网站导航”等类型浏览资源。

### 4.3 站长更新资源

站长在本地项目中新增一条资源数据，补充封面、简介、标签、下载地址和正文说明后，执行构建并推送到静态托管平台完成发布。

## 5. 网站结构需求

### 5.1 页面清单

- 首页
- 资源列表页
- 分类页
- 标签页
- 资源详情页
- 关于页面
- 公告页面
- 友情链接页面（可选）
- 404 页面

### 5.2 导航结构

- 顶部导航包含：首页、分类、标签、公告、关于
- 首页提供分类入口和推荐资源入口
- 每个资源详情页应提供返回列表和相关推荐入口

## 6. 前台功能需求

### 6.1 首页

- 展示网站名称、简介和主视觉区域
- 展示搜索框
- 展示主要分类导航
- 展示推荐资源区
- 展示最新更新资源区
- 展示热门资源区
- 展示站点公告摘要

### 6.2 资源列表页

- 按卡片或列表形式展示资源
- 每条资源展示封面、标题、摘要、标签和更新时间
- 支持分页，或前端按数量分段加载
- 支持按分类、标签和关键字筛选
- 支持按最新、推荐排序

### 6.3 分类页

- 支持按资源大类聚合展示
- 每个分类页应包含分类说明和对应资源列表

### 6.4 标签页

- 支持展示全部标签
- 点击标签后显示对应资源列表

### 6.5 搜索功能

- 使用前端本地搜索实现
- 搜索范围至少包含标题、摘要、标签
- 搜索结果应支持高亮关键词或明显标识匹配项
- 数据规模较小时可直接搜索 JSON
- 数据规模较大时可在构建阶段生成搜索索引文件

### 6.6 资源详情页

- 展示资源标题、摘要、正文说明
- 展示封面图或预览图
- 展示分类、标签、发布日期、更新时间
- 展示资源类型、适用平台、版本号、文件格式、文件大小等信息
- 展示下载按钮或访问按钮
- 若使用网盘链接，应展示提取说明
- 支持展示使用教程、注意事项和更新记录
- 支持展示相关推荐资源

### 6.7 公告与单页

- 提供独立公告页
- 提供关于页面，说明站点定位、资源来源和使用声明
- 提供免责声明或版权说明页

### 6.8 响应式适配

- 网站需兼容桌面端与移动端
- 手机端下导航、搜索和资源卡片需可正常使用
- 图片、按钮和文本排版需适配窄屏

## 7. 内容管理方式

### 7.1 内容来源

网站内容不通过后台录入，而是通过仓库中的静态文件维护。

### 7.2 推荐的数据组织方式

- 使用 `content/resources/` 存放资源详情 Markdown 文件
- 或使用 `data/resources.json` 统一维护资源数据
- 使用 `public/assets/` 存放封面图、截图和图标
- 使用独立文件维护分类、标签、公告和友情链接

### 7.3 推荐内容字段

- `title`：资源标题
- `slug`：页面唯一标识
- `summary`：简短摘要
- `description`：详细说明
- `category`：所属分类
- `tags`：标签数组
- `cover`：封面图路径
- `gallery`：预览图数组
- `type`：资源类型
- `platform`：适用平台
- `version`：版本号
- `fileFormat`：文件格式
- `fileSize`：文件大小
- `downloadUrl`：下载地址
- `backupUrls`：备用地址
- `extractCode`：提取码说明
- `publishedAt`：发布时间
- `updatedAt`：更新时间
- `featured`：是否推荐

### 7.4 内容更新流程

- 站长本地编辑 Markdown 或 JSON 文件
- 本地预览确认内容无误
- 执行构建命令生成静态页面
- 提交到 Git 仓库
- 由 GitHub Pages 或 CI 自动部署上线

## 8. 非功能需求

### 8.1 性能要求

- 页面首屏加载应尽量轻量
- 图片资源需要压缩
- 非首屏图片应支持懒加载
- 大量资源列表页需控制单页展示数量

### 8.2 SEO 要求

- 每个资源详情页需有独立 URL
- 支持页面标题、描述、关键词设置
- 自动生成 sitemap
- 提供 robots.txt
- 支持 Open Graph 和社交分享元信息

### 8.3 可维护性要求

- 内容与页面模板分离
- 分类、标签、公告等结构独立维护
- 文件命名和目录结构清晰
- 便于后续新增页面或切换主题

### 8.4 可用性要求

- 所有外链或下载链接需尽量清晰标识
- 对失效链接应预留“链接可能失效”的静态说明
- 页面需要有 404 页和基础错误提示

### 8.5 兼容性要求

- 支持主流现代浏览器
- 支持移动端浏览器访问

## 9. 视觉与交互要求

### 9.1 设计风格

- 风格以简洁、内容优先、资源可读性强为主
- 首页需要突出搜索和分类入口
- 资源卡片应统一风格，方便快速扫读

### 9.2 交互要求

- 搜索框需始终易于发现
- 分类筛选应简洁直观
- 下载按钮应明显，与普通链接区分开
- 长文详情需有清晰的标题层级和内容间距

## 10. 技术实现建议

### 10.1 推荐方案

- 方案一：`Astro + Markdown/JSON`
- 方案二：`VitePress` 改造为资源展示站
- 方案三：`Vue/Nuxt 静态导出`
- 方案四：`React/Next.js 静态导出`

### 10.2 推荐原则

- 优先选择静态站点生成能力强的方案
- 优先选择对 Markdown 内容支持好的方案
- 优先选择对 GitHub Pages 部署友好的方案

## 11. 明确不做的内容

- 不做登录注册
- 不做后台管理
- 不做数据库存储
- 不做服务端 API
- 不做复杂权限控制
- 不做实时互动功能
- 不做服务端下载统计

## 12. MVP 首版范围

### 12.1 首版必须具备

- 首页
- 分类页
- 标签页
- 资源详情页
- 前端搜索
- 公告页
- 关于页
- 静态资源内容管理方式
- GitHub Pages 级别的部署兼容性

### 12.2 首版可延期

- 友情链接页
- 深度搜索索引优化
- 主题切换
- 多语言
- 第三方评论

## 13. 验收标准

- 网站可在纯静态托管平台正常访问
- 无需数据库和 Node.js 运行服务即可工作
- 用户可完成浏览、筛选、搜索、查看详情、跳转下载的完整流程
- 站长可通过修改本地内容文件完成资源更新
- 网站在桌面端和移动端均可正常浏览
- 每个资源拥有独立详情页与基础 SEO 信息

## 14. 后续扩展方向

- 接入第三方评论系统
- 接入第三方访问统计
- 增加更完整的全文搜索
- 支持多主题切换
- 支持自动从数据文件生成 RSS 或更新日志
